## Вариант 1: Социальная сеть

В системе три сущности: пользователь, стена и сообщения чата. Каждую я вынесла в отдельную коллекцию MongoDB.

## Коллекция `users`

Хранит данные пользователей. Добавила `age` (число) и `interests` (массив строк) чтобы использовать разные типы данных.

```json
{
  "_id": ObjectId("..."),
  "login": "ivan_ivanov",
  "password_hash": "hash123",
  "first_name": "Иван",
  "last_name": "Иванов",
  "age": 28,
  "interests": ["спорт", "музыка", "кино"],
  "created_at": ISODate("2024-01-10T10:00:00Z")
}
```

## Коллекция `wall_posts`

Записи на стене пользователя. У каждого поста два поля-ссылки: `owner_id` - чья это стена, и `author_id` - кто написал (это могут быть разные люди).

```json
{
  "_id": ObjectId("..."),
  "owner_id": ObjectId("..."),
  "author_id": ObjectId("..."),
  "content": "Привет всем!",
  "created_at": ISODate("2024-01-15T12:00:00Z")
}
```

## Коллекция `chat_messages`

P2P-сообщения между пользователями. Ссылки на отправителя и получателя.

```json
{
  "_id": ObjectId("..."),
  "sender_id": ObjectId("..."),
  "receiver_id": ObjectId("..."),
  "text": "Привет! Как дела?",
  "created_at": ISODate("2024-04-01T10:00:00Z")
}
```

## Почему references, а не embedded

Я сначала думала хранить посты прямо внутри документа пользователя — типа массив `wall_posts` в самом документе. Но потом поняла, что это не работает: на стену пишут разные люди, не только сам владелец. Если хранить пост внутри одного пользователя - непонятно в чей документ его класть. Плюс постов может накопиться очень много, документ будет постоянно расти. Поэтому сделала отдельную коллекцию с `owner_id` и `author_id`.

С сообщениями похожая проблема - каждое сообщение связывает двух пользователей. Если хранить их у отправителя, то получатель не найдёт входящие без лишних запросов. Если у получателя - тогда у отправителя нет истории исходящих. Единственный нормальный вариант - отдельная коллекция со ссылками на обоих.

| Связь | Тип | Причина |
|---|---|---|
| Пост → Владелец стены | Reference (owner_id) | Один пост - одна стена, но автор может быть другим |
| Пост → Автор | Reference (author_id) | Автор и владелец стены - разные люди |
| Сообщение → Отправитель | Reference (sender_id) | Сообщение принадлежит двум пользователям сразу |
| Сообщение → Получатель | Reference (receiver_id) | Сообщение принадлежит двум пользователям сразу |
