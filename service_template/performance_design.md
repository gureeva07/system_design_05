# Проектирование кеширования и rate limiting

**Вариант 1 - Социальная сеть**

---

## 1. Анализ производительности

Я посмотрела на все эндпоинты и попыталась понять, какие из них будут вызываться чаще всего и где могут быть проблемы с производительностью.

**Что вызывается часто (hot paths):**
- Поиск пользователя по логину - это происходит каждый раз когда кто-то открывает чужой профиль
- Загрузка стены - пользователи постоянно обновляют ленту
- Получение сообщений - тоже читается очень часто

**Где может быть медленно:**
- Поиск по имени (`?name=`) - это regex-запрос к MongoDB, он не использует индекс и делает полный скан коллекции. Если пользователей тысячи - это будет медленно
- Загрузка стены - каждый раз запрос к MongoDB, даже если с момента последнего чтения ничего не изменилось
- Поиск по логину - то же самое, данные меняются редко, а запросы идут постоянно

**Требования к производительности (примерные):**
- Поиск пользователя: хотелось бы < 100 мс
- Загрузка стены: < 200 мс
- Под нагрузкой система должна выдерживать хотя бы несколько сотен запросов в секунду на чтение

## 2. Стратегия кеширования

### Что решила кешировать

Я выбрала два эндпоинта:

**Поиск пользователя по логину** - профиль пользователя меняется очень редко (в нашем API вообще нет эндпоинта для его изменения), а читается постоянно. Идеальный кандидат для кеша.

**Загрузка стены** - стена тоже читается намного чаще, чем обновляется. Пользователь может обновить страницу 10 раз за минуту, а новый пост появится один раз.

Сообщения я решила не кешировать - там важна актуальность, человек ждёт новых сообщений в реальном времени.

### Какую стратегию выбрала

Я выбрала **Cache-Aside (Lazy Loading)**. Работает так:
1. Пришёл запрос → смотрим в кеш
2. Если данные есть (cache hit) → отдаём из кеша, в MongoDB не идём
3. Если данных нет (cache miss) → идём в MongoDB, получаем результат, кладём в кеш и отдаём клиенту

Почему Cache-Aside, а не что-то другое:
- **Read-Through** требует специальной библиотеки или прокси, которая сама умеет ходить в БД - у нас этого нет
- **Write-Through** означает, что при каждой записи нужно обновлять и кеш. Но посты добавляются редко, проще просто сбросить кеш
- **Write-Back** - слишком сложно для нашего случая, данные могут потеряться если кеш упадёт до записи в БД

Cache-Aside - самый простой вариант, который легко понять и реализовать.

### TTL

| Что кешируем | Ключ | TTL | Почему такой TTL |
|---|---|---|---|
| Профиль пользователя | `user:login:{login}` | 5 минут | Данные меняются очень редко, можно держать дольше |
| Стена пользователя | `wall:{user_id}` | 1 минута | Посты добавляются чаще, большая задержка была бы заметна |

### Инвалидация кеша

Для стены я сделала явную инвалидацию: когда пользователь добавляет новый пост через `POST /wall/{id}`, кеш для этой стены сразу удаляется. Следующий GET-запрос получит свежие данные из MongoDB.

Без этого получилась бы неприятная ситуация: пользователь написал пост, обновил страницу, а поста нет - он ещё не появился потому что кеш не обновился. Так делать нельзя.

Для профиля пользователя инвалидации нет - только по TTL. Потому что в нашем API профиль вообще нельзя изменить через эндпоинты.

### Реализация

Кеш написала сама в `src/Cache.hpp` и `src/Cache.cpp` - это простой словарь `key → {значение, время_истечения}` с mutex для потокобезопасности. В реальном проекте вместо этого использовали бы Redis - он умеет TTL сам, не теряет данные при перезапуске и работает между несколькими серверами. Но для учебного проекта in-memory вполне достаточно.

## 3. Rate Limiting

### Почему это нужно

Поиск по имени (`?name=Ivan`) - это regex по всей коллекции MongoDB. Если кто-то начнёт слать эти запросы в цикле (случайно или специально), MongoDB будет постоянно делать дорогие операции и остальные запросы начнут тормозить. Rate limiting защищает от этого.

### Какой алгоритм выбрала

Я выбрала **Fixed Window Counter** - это самый простой алгоритм.

Как работает: у каждого пользователя есть счётчик запросов. Счётчик живёт в рамках временного окна (например, 1 минута). Если счётчик не превысил лимит - запрос проходит, счётчик увеличивается. Если превысил - возвращаем 429. Когда окно заканчивается - счётчик обнуляется.

Почему не другие алгоритмы:
- **Token Bucket** - более гибкий (позволяет кратковременные всплески), но сложнее в реализации: нужно хранить время последнего пополнения и дробный счётчик токенов
- **Sliding Window Log** - самый точный, но хранит метку времени для каждого запроса. При большом трафике это много памяти
- **Leaky Bucket** - хорошо сглаживает нагрузку, но тоже сложнее Fixed Window

Для учебного задания Fixed Window - оптимальный выбор.

### Лимиты

| Эндпоинт | Ключ для счётчика | Лимит | Окно |
|---|---|---|---|
| `GET /users?name=` | user_id пользователя | 30 запросов | 60 секунд |

30 запросов в минуту - это нормально для реального использования (поиск по имени не нужен чаще), но достаточно мало чтобы защититься от злоупотреблений.

### HTTP-заголовки

При каждом запросе к `?name=` возвращаю заголовки:
- `X-RateLimit-Limit: 30` - максимальный лимит
- `X-RateLimit-Remaining: 27` - сколько запросов осталось в текущем окне
- `X-RateLimit-Reset: 1746356400` - unix-время когда окно сбросится

Это стандартная практика - клиент видит сколько у него осталось запросов и когда можно снова стучаться.

При превышении:
```
HTTP 429 Too Many Requests
{"error":"Rate limit exceeded. Try again later."}
```

## 4. Как это влияет на производительность

### Кеширование

Без кеша каждый запрос на поиск по логину или загрузку стены идёт в MongoDB. С кешем большинство запросов отвечает из памяти.

В тесте я замерила время:
- Первый запрос (cache miss, идёт в MongoDB): **34 мс**
- Второй запрос (cache hit, из памяти): **14 мс**

Ускорение в 2.4 раза. И это при том что MongoDB запущена локально - в реальном окружении сетевая задержка до базы данных была бы больше, и эффект от кеша был бы заметнее.

Если предположить что стену одного пользователя читают 100 раз в минуту, а обновляют 1 раз:
- Без кеша: 100 запросов к MongoDB
- С кешем (TTL=60с): ~2 запроса к MongoDB (один при первом чтении, один после инвалидации)

**Hit rate** (процент ответов из кеша) в таком сценарии ≈ 98%.

### Rate limiting

Rate limiting не ускоряет систему напрямую - он защищает её от перегрузки. Без него один клиент мог бы отправить тысячи дорогих regex-запросов и положить MongoDB для всех остальных. С rate limiting каждый клиент ограничен 30 запросами в минуту.

### Метрики для мониторинга

Чтобы понять насколько хорошо работает кеш, нужно смотреть на:
- **Cache hit rate** = число попаданий / (попадания + промахи). Чем выше - тем лучше. Хороший показатель - выше 80-90%
- **Среднее время ответа** по каждому эндпоинту - должно упасть после включения кеша
- **Количество 429 ответов** - показывает насколько часто кто-то упирается в лимит

Я реализовала сбор метрик через эндпоинт `/metrics` в формате Prometheus. В `Cache::Get()` два атомарных счётчика - один для хитов, другой для промахов. В `RateLimiter::IsAllowed()` счётчик заблокированных запросов. Prometheus опрашивает `/metrics` каждые 5 секунд, Grafana строит графики в реальном времени.

В итоге на дашборде видно: hit rate 0.97 (97% запросов из кеша), промахи только в начале пока кеш прогревается, и счётчик заблокированных запросов rate limiter.
